ECU Simulation with Yocto(Poky)

In this personal project, I aimed to simulate various ECUs of a car, from the central MCU to various ECUs with sensors. I implemented ECUs for doors, battery, engine, and even a LiDAR and a traffic sign recognition camera. The goal of this project was to enable communication between different ECUs using the CAN protocol and embed the software in an image of personalized Linux distro using Poky.

The code is entirely written in C++ and uses the SocketCan library. I also implemented a console-based graphical interface that simulates the car's dashboard, displaying open or closed doors, speed, temperature, and engine status. The graphical interface also shows the environment recorded by the LiDAR (with randomly generated points).

The different ECUs communicate with each other via CAN and sockets, so the LiDAR is not functional while the car is stopped, speed does not increase when the car is stationary, and if obstacles are detected, the car stops. Additionally, when a traffic sign is detected with a certain speed, the MCU sends messages via CAN to reduce the speed, which in turn sends a request to the ECU to adjust the speed.

Several cases of ECU communication are implemented in the program, with each being independent and communicating only through CAN.

Also, to integrate the binaries into the Linux image, as I mentioned, I used Poky. I created a layer where I added the C++ code as well as a Linux service that starts at boot and sets up the virtual CAN interfaces to which each ECU connects. After building, the image was launched on QEMU, where the simulation ran successfully.


Technologies used:

C++, Makefile, Yocto, QEMU, SocketCAN/CAN protocol

5 DoF Robot Arm


This is one of the most complex projects I have created, and it is my bachelor's thesis for my robotics degree.

Mechanical and Design Aspect

Everything was done by me, and all the parts and mechanisms were designed from scratch using SolidWorks. I also selected the motors and other necessary components for the robot.

Software Components

The software is divided into three main sections:

1. Control System

Written in C++ using Arduino. I tried to avoid using libraries, so the simultaneous motor control is handled through microcontroller interrupts to generate steps for the motors, creating the illusion of synchronized movement. An algorithm calculates acceleration for each motor and sets the next interrupt accordingly.

2. Inverse Kinematics

Written in Python, this part calculates the angle of each joint based on a given point in 3D coordinates.

3. User Interface

Written in C#, this interface communicates with the control system, allowing the user to input the desired position for the robot or manually set the angle for each joint.


Technologies used:

C++(Arduino), C#(WindowsForm), Python(PyGame), SolidWorks, 3DPrint

PCB for Controller Step By Step Motor

This project was carried out as part of the Bosch scholarship program that I participated in during my fourth year of university.

In this project, I chose to make a controller for a stepper motor. For this, I divided it into several parts: first, I had the voltage control part with PWM, the phase shift part (for the motor steps) where I used a 555 timer and a multiplexer, the power part where I used 2 H-bridges, and other minor components.

I tested everything on software, designed it in KiCad, and then physically ordered a PCB board and electrical components to test them.


Technologies used:

KiCAD
